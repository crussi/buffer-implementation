## Safety Issues

| # | What to Fix | Recommended Change | Where |
|---|---|---|---|
| 1 | **Double-free of `text` pointer** — `history_undo`/`history_redo` push an `Action` onto the stack then call `free_action(&a)`, freeing the pointer the stack now holds | Do not call `free_action(&a)` after a successful `push_action`; the stack owns the pointer from that point | `editor_history.c` |
| 2 | **No bounds check on `row` before array access** — `editorInsertChar`, `editorDeleteChar`, `editorInsertCR`, `editorDeleteCR` all index `buf->rows[row]` without validating `row` is within `[0, numrows)` | Add a guard: `if (!e || row < 0 || row >= e->buf->numrows) return;` at the top of each function | `editor.c` |
| 3 | **`col` read before validation in `editorDeleteChar`** — `line[col]` is read to capture the deleted character before any bounds check | Validate `col` is within `[0, row.length)` before reading `line[col]` | `editor.c` |
| 4 | **`free_action` is public but callers of `peek_action` must not call it** — exposes a footgun since `peek_action` does not transfer ownership | Make `free_action` `static` in `action_stack.c` and remove it from `action_stack.h`; add an `action_clone()` for callers that need an owned copy | `action_stack.h/.c` |
| 5 | **Memory leak in `fileToBuf` on partial failure** — if a mid-loop `malloc` fails, previously allocated `row.line` pointers are never freed before returning `NULL` | Extract a `free_buf_rows(buffer *buf)` helper and call it in all early-exit failure paths | `buffer.c` |

This is a solid foundation for a text editor — the layered architecture (buffer → action_stack → editor_history → editor) is well-conceived and the memory management in action_stack.c is especially careful. Here's the review across your five questions:

---

## 1. Missing Editing Functionality

The core primitives are there, but a real editing session will quickly need:

**Cursor management** — there's no cursor state. Right now `row`/`col` are passed in on every call, meaning the caller has to track position externally. An `EditorCursor` struct embedded in `Editor` would centralize this and let you write `editorMoveCursorLeft()`, `editorMoveCursorRight()`, etc.

**Selection / region operations** — cut, copy, paste. These require knowing a start and end position and operating on a range of text, which your current action types don't model.

**Bulk text insertion** — your `text` field in `Action` is allocated but never used. There's no `editorInsertString()` that inserts multiple characters as a single undoable action. Typing a word and pressing undo should ideally undo the whole word, not one letter at a time.

**Save/load from a named file** — `editor_new_from_file` takes an already-open `FILE*`, but there's no `editor_save(Editor *e, const char *path)`. The editor has no concept of its own filename.

**Search** — find (and ideally find/replace) is almost always considered a baseline feature.

---

## 2. Level of Abstraction

The layering is genuinely good: `buffer.c` knows nothing about history, `editor_history.c` knows nothing about the editor struct, and `editor.c` is the only place they meet. That's the right approach.

A few refinements worth considering:

**`main.c` bypasses the editor layer entirely**, calling `insertChar`/`deleteChar` directly on the buffer. Once `editor.c` exists, `main.c` should only interact through `Editor*`. This is more of a test-harness issue than a design flaw, but worth noting.

**`history_undo` and `history_redo` take a raw `buffer*`**, coupling `editor_history` to `buffer.h`. Since the history already lives inside an `Editor`, it would be cleaner to pass `Editor*` through or use a callback, though this is a minor point for a project of this size.

**`editorDeleteChar` doesn't bounds-check `col`** before reading `e->buf->rows[row].line[col]`. If `col` is out of range you get undefined behavior before the action is even recorded.

---

## 3. Missing Functionality in buffer.c (your constraint noted)

Since you asked for an opinion: the main gap is the absence of a **range/substring API**. Operations like "get the text from (r1,c1) to (r2,c2)" or "replace a span" would be useful for implementing selection-based editing, search-and-replace, and smarter undo grouping. Currently any multi-character operation has to be assembled from single-char calls. A `getSubstring(buffer*, Position, Position)` returning a heap-allocated string would bridge that gap cleanly.

Also, `fileToBuf` has a subtle memory leak: if any `malloc` for `buf->rows[buf->numrows].line` fails mid-loop, the already-allocated row lines are never freed before returning NULL.

---

## 4. Safety Issues

**Bounds checking gaps in editor.c:**
- `editorInsertChar` and `editorDeleteChar` index directly into `e->buf->rows[row]` without checking that `row` is within `[0, buf->numrows)`. An out-of-bounds `row` is undefined behavior.
- `editorDeleteChar` reads `line[col]` before validating `col`.

**`free_action` is declared in the header but not `static`** — it's part of the public API, but `peek_action` explicitly documents that the caller must *not* free the returned action's text. Exposing `free_action` publicly creates a footgun. Consider making it static and providing a `action_copy()` function instead if callers need owned copies.

**`history_undo` and `history_redo` call `free_action(&a)` after `push_action(h->redo_stack, a)`** — but `push_action` copies the `Action` struct by value, meaning both the stack and `a` hold the same `text` pointer. Calling `free_action(&a)` then frees the pointer the stack is holding. This is a use-after-free / double-free bug. Since `text` is currently always NULL in practice it doesn't crash yet, but it will when you use the `text` field.

**`fileToBuf` leak on failure** — noted above, rows allocated before a mid-loop failure are not freed.

**`action_stack.c`'s `free_action` is not `static`** despite being listed as an "internal helper" in the comment — it ends up in the public header too. Pick one.

---

## 5. Naming Conventions

The inconsistency is the main issue. `buffer.c` uses camelCase (`newBuf`, `insertChar`, `fileToBuf`) while everything else uses snake_case (`new_action_stack`, `editor_new_empty`, `history_record`). In C, snake_case for functions and variables is idiomatic (following the standard library's style), so the buffer naming is the odd one out.

Specific notes:
- `row` as a type name is fine but very generic; `BufferRow` or `TextRow` would be clearer at a glance.
- `nread` in `fileToBuf` is typed `long int` but used as a length/index — `ssize_t` or `size_t` would be more idiomatic.
- `newCap` / `newCapacity` appear in the same file (`buffer.c`) doing the same job with different names; pick one style.
- `h` as a parameter name for `EditorHistory*` is fine locally but `hist` would be slightly more readable.
- The commented-out old `editorDeleteCR` should be removed rather than left in.

---

**Summary priority list** if you're deciding what to fix first:

1. The double-free bug in `history_undo`/`history_redo` with `text` payloads
2. Bounds checking before array access in `editor.c`
3. The `fileToBuf` leak on partial failure
4. Unify naming to snake_case across all files
5. Add cursor state to `Editor` before building any UI layer
